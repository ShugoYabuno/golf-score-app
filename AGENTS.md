# AGENTS.md

このファイルは、このリポジトリで作業するエージェント向けの実装ルールです。  
UI/UX方針は `.cursor/rules/golf-score-ui-ux.mdc` と同一意図で運用します。

## Product Intent

- ゴルフスコア記録を「速く、迷わず、少し楽しい」体験にする。
- Duolingoのような達成感は参考にするが、毎日利用前提の仕組みは持ち込まない。
- 最優先はラウンド中の入力摩擦を減らすこと。
- ゲーミフィケーションは入力体験を邪魔しない補助として扱う。

## Core Principles

- Fast: 1ホール入力10秒以内、主要入力3タップ以内を目標にする。
- Safe: 誤入力を減らし、操作後の修正を軽くする。
- Minimal: MVPは入力・履歴・分析を中心にする。
- Encouraging: 良いスコアだけでなく記録した行為自体を称える。
- Comparable: 18H/9H/ショート、コース、ティーなど比較条件を混ぜない。
- Delightful: 完了時や改善時に軽い達成感を返す。

## Input Rules

- 最頻出操作は画面下部サムゾーンに置く。
- タップターゲットは最低48x48pt、主要ボタンは52pt以上を目安にする。
- スコア入力は選択式UIを優先し、テキスト入力を避ける。
- ホール番号、Par、進捗は常に見える状態にする。
- 詳細項目は設定でON/OFF可能にする。
- 入力確認ダイアログは多用しない。必須不足はその場で明示する。
- 保存は非ブロッキングで行う。保存状態は簡潔に表示する。
- ラウンド中は入力テンポを最優先し、修正・補助設定・詳細記録は主導線を妨げない位置に置く。
- ユーザーが入力しなくても推定・計算できる情報は、可能な限り追加入力にしない。

## UI Review Mindset

- エージェントはUI変更時に、実装だけでなく実際の使用状況を想像して画面をレビューする。
- 特にラウンド中の画面は、屋外・片手・短時間・次のショット前という文脈で確認する。
- 画面を見て「主目的が何か」「次に押す場所が自然に分かるか」「余計な情報が入力を邪魔していないか」を自律的に点検する。
- 主要操作がスクロールや折りたたみの奥に隠れていないかを確認する。
- UIが大きすぎて必要情報を押し出していないか、小さすぎて誤操作を誘発しないか、密度と操作性のバランスを見る。
- 同じ画面内に異なる利用モードが混ざっていないかを確認し、必要ならモード分離・段階開示・一時的な重ね表示を提案する。
- 追加記録や分析項目は、ユーザーの入力負荷に見合う価値があるかを確認する。
- 既存データから算出できる項目、ラウンド後に見ればよい項目、入力中に不要な項目は、入力画面から外す候補として扱う。
- 改善提案は「何を消すか」「何を隠すか」「何を後回しにするか」も含めて考える。

## Gamification Rules

- 記録を妨げない範囲でゲーム要素を使う。
- ホール完了、ラウンド完了、ベスト更新、初バーディ、目標達成に軽い達成演出を返す。
- ラウンド中の演出は控えめにし、XP・バッジ・レベルアップの表示は完了後に寄せる。
- 演出は最大2秒以内を目安にする。
- XP/レベルは上達可視化のために使い、催促目的で使わない。
- バッジは段階的に開放し、未取得を大量表示しない。
- 採用: XP、レベル、バッジ、改善ロードマップ、年間振り返り、同レベル比較。
- 不採用: 毎日ストリーク、デイリーゴール、ライフ制、催促型通知、無差別ランキング。

## Tone And Copy

- アプリは審判ではなく伴走するコーチとして振る舞う。
- 悪いスコアの日でも最初に記録完了を肯定する。
- 責める表現（悪化、平均以下、失敗、遅れています）を使わない。
- 改善は相対値でも伝える（例: 先月比、直近5ラウンド比）。
- 弱点提示は「次に意識するポイント」として前向きに示す。

## Visual Rules

- 明るく前向きで親しみのあるUIにする。
- 良い結果は祝う色、悪い結果は淡い警告色に留める。
- 1画面1主目的を守る。特にラウンド中は入力以外を増やさない。
- 主要CTAは下部配置を優先する。
- 状態は色、アイコン、短いアニメーションで伝える。
- キャラクターは評価者ではなく伴走者として使う。

## Screen-Specific Rules

### Home

- 次アクション（開始・再開）がすぐ分かる構成にする。
- 直近ラウンド、短期推移、最近バッジを少量だけ見せる。
- 未達成を赤で強く煽らない。

### Score Input

- 現在ホール、Par、進捗、入力、次へに集中する。
- 入力に必要な情報と操作が一画面内で把握できる状態を目指す。
- 主要入力は片手で素早く微調整でき、誤操作時も戻しやすいUIを優先する。
- 詳細入力は任意・設定依存にし、主入力の流れを押し下げたり分断したりしない。
- ラウンド中に必要ない情報や、別モードで扱うべき情報は入力画面から切り離す。
- ラウンド中にバッジ/ランキング/チャレンジ更新を割り込ませない。

### Round Complete

- この画面を感情的ピークにする。
- スコア、ハイライト、獲得報酬、次回目標を提示する。
- ベスト更新でなくても肯定的な完了体験にする。
- 完了直後に詳細統計へ強制遷移しない。

### History / Analytics

- 数字羅列で終わらせず、自然言語コメントを添える。
- ベスト、平均、目標、先月比など文脈ある比較を優先する。
- 比較条件が混ざる表示を避ける。
- 弱点は次アクションとして前向きに提示する。

### Goals / Challenges

- 目標はユーザー主導で設定できるようにする。
- 期間は月間やラウンド単位など実利用頻度に合わせる。
- 未達成を責めず、挑戦と継続を肯定する。
- 比較は友人内/同ハンデ帯を優先する。

### Profile / Level

- 優劣より成長の軌跡を見せる。
- 通算指標と獲得実績を整理して表示する。
- 未取得バッジの空欄を大量表示しない。

## Anti-Patterns

- ラウンド中に複雑なフォーム入力を要求する。
- 主要操作が画面外、折りたたみ内、または補助UIの下に隠れる。
- 主目的の異なるナビゲーションや情報を、ラウンド中の入力画面に常設する。
- 既存入力から計算・推定できる項目を、追加記録としてユーザーに入力させる。
- 情報量やボタンサイズの都合で、実際の利用テンポを落とす。
- 毎日利用前提の仕組みを導入する。
- 成績不振を責める文言や強い警告色で圧をかける。
- 演出が入力テンポを止める。
- 初回設定を一気に要求する。
- 無差別グローバルランキングを主役にする。

## Implementation Checklist

- 主要操作は片手親指で届くか。
- 1ホール入力は10秒以内に収まりそうか。
- 主要操作と必要情報は、ひと目で把握できる範囲に収まっているか。
- 小さいタップ領域やテキスト入力に依存していないか。
- 詳細入力や補助導線が、主入力の流れを邪魔していないか。
- 入力させている項目は、本当にその場でユーザーが記録する価値があるか。
- 算出できる情報を入力させていないか。
- 異なる利用モードが同じ画面で競合していないか。
- ゲーム要素がラウンド入力を妨げていないか。
- 悪いスコアの日も肯定的な体験になっているか。
- 比較条件が混ざっていないか。
- 画面の主目的は1つに絞れているか。
- 完了時に軽い達成感を返せているか。
